# 中文科技论文章节地图

当任务主要卡在“这一段到底该写成哪个章节的样子”时，先读本文件。它不是给人类作者看的泛泛提纲，而是给模型做章节分流和段落起草的操作说明。

## 路由规则

按真实任务选择入口：

| 用户要做什么 | 先读哪里 | 再读哪里 |
| --- | --- | --- |
| 起草某一章节 | 对应章节条目 | [style-controls-zh.md](style-controls-zh.md) |
| 改写某一章节但不改变原意 | 对应章节条目中的“构建顺序”和“常见风险” | [revision-checklist-zh.md](revision-checklist-zh.md) |
| 诊断某一段为什么不像论文 | 对应章节条目中的“读者目标”与“常见风险” | [style-controls-zh.md](style-controls-zh.md) |
| 把笔记、图表或要点扩成成段文字 | 对应章节条目的“句子功能” | [style-controls-zh.md](style-controls-zh.md) |

真正落笔前先确定四件事：

1. 当前任务是 `起草`、`改写`、`诊断`、`压缩` 还是 `扩写`。
2. 目标章节是什么，不能拿 Discussion 的写法去写 Results。
3. 可用素材是什么，哪些数字、比较、术语、图表编号必须保留。
4. 当前段落要消除读者的哪个疑问，不能只按作者记忆顺序堆信息。

## 引言

### 读者目标

让读者明白：这个问题为什么重要、前人做到哪里、真正缺的是什么、本文为什么值得出现。

### 构建顺序

1. 先立住问题的重要性或背景。
2. 再把已有研究按功能分组，而不是按文献编号排队。
3. 接着指出空白、争议、局限或未解问题。
4. 最后引出本文目标、方法和贡献。

### 句子功能

常见功能顺序如下：

- 说明主题价值，
- 缩小到具体问题，
- 概括前人研究版图，
- 指出缺口或矛盾，
- 交代本文目标，
- 点出本文做法或贡献。

### 起草骨架

优先使用：

`问题重要性 -> 已有认识 -> 尚未解决之处 -> 本文工作`

可直接对自己下这样的指令：

- `不要把文献写成流水账，先分组再概括。`
- `在空白没有立住之前，不要急着写“本文提出”。`
- `把背景改成服务于当前研究问题的跑道，而不是大而泛的教科书导言。`

### 常见风险

- 开头过窄，读者还没被带进来就进入技术细节。
- 前人研究逐篇列举，却没有形成研究地图。
- “本文工作”出现得太早，像硬插广告。
- 用“首次”“新型”等词，但前文并未建立缺口。

## 方法

### 读者目标

让读者知道做了什么、为什么这样做、哪些步骤借鉴前人、哪些约束会影响后文解释。

### 构建顺序

1. 先给总体设计、材料、对象或装置全景。
2. 再按可复现顺序写关键步骤。
3. 说明哪些方法沿用前人，哪些地方做了修改。
4. 只在影响理解的地方补“为什么这样选”。
5. 如果方法存在局限或条件约束，要在这里说清楚。

### 句子功能

- 说明设计或来源，
- 交代对象、材料、设备，
- 描述步骤顺序，
- 解释关键选择，
- 标出方法来源，
- 说明修改点，
- 交代约束或局限。

### 起草骨架

优先使用：

`总体设计 -> 关键步骤 -> 前人方法关系 -> 选择理由或约束`

如果输入是实验笔记而不是成文段落，先抽取这几个槽位：

- 做了什么，
- 按什么顺序做，
- 用了什么材料或仪器，
- 为什么这样做，
- 哪一步来自前人方法。

### 常见风险

- 写成实验流水账，没有章节内部结构。
- 把背景原理和操作细节揉成一团。
- 无主语或被动表达过多，分不清本文操作与通行做法。
- 对每个细节一视同仁，真正重要的选择反而埋住了。

## 结果

### 读者目标

让读者看见最重要的发现、理解关键数据意味着什么，并知道哪些解释还不能在这一节说满。

### 构建顺序

1. 先把读者拉回当前比较或当前图表要回答的问题。
2. 先报主结论，再给支撑它的数字或差异。
3. 需要时补充与预期、模型或前人结果的关系。
4. 有异常、边界条件或技术限制时一起交代。
5. 只做克制解释，不提前把结果段写成讨论段。

### 句子功能

- 设定当前比较对象，
- 报告主结果，
- 给关键数字或趋势，
- 做必要比较，
- 指出异常或边界，
- 给出保守意义。

### 起草骨架

优先使用：

`当前问题或图表焦点 -> 主结果 -> 关键数字 -> 异常或边界 -> 保守解释`

图表驱动写作时，按这三个问题组织：

1. 读者先应该从图表里看到什么？
2. 哪些数字或比较能证明这一点？
3. 哪个异常或限制让结论保持诚实？

### 常见风险

- 只报数据，不告诉读者为什么重要。
- 把图表里每个点都在正文里复述。
- 提前进入机制解释或价值判断。
- 使用“显著”“明显”“仅”“高达”等词，却没有给出足够上下文。

## 讨论

### 读者目标

解释结果意味着什么、本文相对前人做出了什么推进、结论边界在哪里。

### 构建顺序

1. 开头重述本文回答了什么问题或实现了什么。
2. 解释主要发现的含义。
3. 明确写出与前人研究的关系。
4. 交代局限，并说明这些局限如何影响解释。
5. 收束到意义、应用或未来工作。

### 句子功能

- 重申答案，
- 解释结果，
- 对照前人研究，
- 说明一致或不一致的原因，
- 交代局限，
- 提出意义、应用或下一步。

### 关系标签

比较前人工作时，优先明确使用以下关系而不是含糊带过：

- `支持`,
- `扩展`,
- `挑战`,
- `互补`,
- `限定`,
- `仍未解决`.

### 常见风险

- 只是把结果再说一遍，没有新增解释。
- 证据不足时把因果或普适性说得太满。
- 先下绝对判断，最后例行补一句“仍有局限”。
- 与前人研究比较时不说关系，只说“与已有研究一致/不同”。

## 结论

### 读者目标

在离开论文前，让读者准确记住本文做成了什么、意义是什么、边界在哪里。

### 构建顺序

1. 简要回到研究对象或目标。
2. 重述核心结果或贡献。
3. 点出意义、应用或理论价值。
4. 只有在能帮助收束时才补局限或未来方向。

### 句子功能

- 回顾问题，
- 指出贡献，
- 说明意义，
- 说出实际或概念层面的 takeaway，
- 划出剩余边界。

### 常见风险

- 与摘要机械重复。
- 在结论里突然塞入新证据。
- 结尾过空，像套在任何论文上都成立。

## 摘要

### 读者目标

即使读者只看摘要，也能明白研究问题、做法、主要结果和价值。

### 构建顺序

1. 快速交代背景与研究必要性。
2. 说明本文做了什么。
3. 简写关键方法。
4. 写出主结果并带解释。
5. 用一句话点出贡献、意义或应用方向。

### 句子功能

- 建立背景，
- 点出需求或空白，
- 交代方法，
- 报告主发现，
- 说明为什么重要。

### 压缩规则

- 方法只保留判断是否值得阅读全文所需的信息。
- 必须让读者看清哪些内容属于本文工作。
- 数字不能裸报，要配评价或方向。
- 篇幅紧时可以合并功能，但不能丢失“问题 -> 做法 -> 结果 -> 意义”的逻辑链。

### 常见风险

- 像一串并列事实，没有内在逻辑。
- 数字很多，但没有告诉读者高低、增减或意义。
- 常规方法写太细，主结果反而模糊。
- 已有事实与本文发现混在一起。

## 标题

### 读者目标

让目标读者一眼知道论文写的不是哪个大领域，而是这篇论文具体贡献了什么。

### 构建顺序

1. 找出论文真正的研究对象。
2. 找出方法、对象、场景或贡献中最值得暴露的一项。
3. 只保留对目标读者识别这篇文章有帮助的词。

### 标题检查

- 标题是否暴露了对象、方法或结论角度？
- 缩写对目标读者是否真的安全？
- 是否出现了过密的名词堆积？
- “新型”“首次”“高效”等词是否真的有证据支撑？

### 常见风险

- 像临时工作标题，只报研究领域。
- 只写主题，不写本文产出。
- 名词链太长，读起来像标签堆叠。
- 轻率使用高强度宣传词。

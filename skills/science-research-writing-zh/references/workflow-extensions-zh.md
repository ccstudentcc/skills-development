# 中文科研写作流程与扩展场景

当任务不再只是“把这一段改顺”，而是涉及整篇写作顺序、图表与结果联动、审稿回复、综述文章或大众摘要时，读这个文件。

## 快速导航

- 如果还没看清任务属于哪类扩展场景，先看 `路由规则`。
- 处理整篇推进时，跳到 `论文装配顺序` 或 `起草流程`。
- 处理图表驱动写作时，跳到 `图表、表格与结果叙事`。
- 处理相邻科研表达任务时，跳到 `审稿回复`、`综述文章扩展` 或 `大众摘要扩展`。

## 路由规则

| 用户要做什么 | 先读哪里 |
| --- | --- |
| 从笔记、图表或提纲推进成完整论文写作 | `论文装配顺序` 与 `起草流程` |
| 优化图表、表格、图例或结果段的联动 | `图表、表格与结果叙事` |
| 根据审稿意见修改并撰写回复 | `审稿回复` |
| 写综述文章而不是原始研究论文 | `综述文章扩展` |
| 把科研内容改成大众可读摘要 | `大众摘要扩展` |

## 联动技能路由

当任务已经从“把科研文字写清楚”延伸到相邻工作流时，不要勉强让本技能包办到底。

- 需要围绕整篇文档做分阶段协作、分节推进和读者视角复查时，转用 `anthropics/skills` 仓库中的 `doc-coauthoring`。
- 需要产出概念图、框架图、图形摘要或其他精修后的单页视觉稿时，转用 `anthropics/skills` 仓库中的 `canvas-design`。
- 需要补文献、核对引文信息、生成 BibTeX 或清理参考文献条目时，转用 `K-Dense-AI/claude-scientific-writer` 仓库中的 `citation-management`。
- 论证结构已经稳住，只剩下去除机器腔、让中文更自然时，再转用 `op7418/Humanizer-zh` 仓库中的 `humanizer-zh`。
- 这里不要写绝对路径：有安装时模型会按技能名自然检索，没有这些技能时要明确告诉用户，并改用本技能里最接近的 fallback，不要默认该技能一定能调用。

## 缺失技能时的 fallback 映射

只有在对应联动技能当前不可用时，才使用下面这些回退方案。

- `doc-coauthoring` 不可用：仍在本技能内完成，但要显式拆成“澄清任务、分节规划、起草、精修”四步，不要假装自己处在完整的分阶段协作流程里。
- `canvas-design` 不可用：不要承诺直接产出成熟图稿，改为提供图示说明书，包括图面核心信息、版式层次、各分区意图和 caption 草稿，供后续视觉流程执行。
- `citation-management` 不可用：只整理、核对或规范化用户已经提供的引文信息，不能补造 DOI、BibTeX 字段、作者信息或缺失文献。
- `humanizer-zh` 不可用：改用 `style-controls-zh.md` 和 `revision-checklist-zh.md` 做最后一轮去机器腔精修，重点看句式重复、节奏过平、判断词过满和连接词套路化。

## 论文装配顺序

整篇文章尚未成形时，不要随便决定章节写作顺序。

优先使用以下顺序：

1. 先打磨图表和表格。
2. 再据此写结果。
3. 然后写方法。
4. 等研究故事更清楚后写引言。
5. 在结果逻辑稳定后写讨论。
6. 最后写摘要和标题。

这样做的原因：

- 图表和表格最先暴露研究真正的故事骨架。
- 结果段会因为“每张图表各讲一个点”而更容易写。
- 引言更容易围绕真实贡献来收束。
- 摘要和标题只有在全文逻辑稳定后才容易写准。

## 起草流程

把写作分成三个任务，不要混在一起。

### 1. 写前准备

真正写句子前先完成：

- 收集并整理素材，
- 明确 take-home message，
- 搭好到段落粒度的路线图，
- 先把同类信息放在一起，再考虑如何成句。

如果输入是零散笔记，先整理出：

- 章节目标，
- 段落目标，
- 必须保留的证据，
- 每句承担什么功能，
- 哪些地方需要保留边界或局限。

### 2. 初稿

初稿主要解决逻辑，不追求局部漂亮。

- 先把意思按完整句写出来，
- 先保证组织顺序，再做修辞优化，
- 不要一边起草一边强迫自己做细修，
- 只要段落功能清楚，就继续往前推进。

### 3. 修订

大部分质量提升发生在修订阶段。

建议按这个顺序做：

1. 大声读一遍。
2. 标出每句主谓，检查动词是否有力、主动、没有被埋住。
3. 删除赘语，尤其是空洞开场、弱化副词和名词化表达。
4. 给每段贴一句“主旨标签”，必要时调段序。
5. 最后核对数字、引用和跨章节一致性。

## 图表、表格与结果叙事

图表和表格不是附属品，而是研究故事的骨架。

### 核心规则

- 每张图或每个表尽量只承担一个清楚的点。
- 图题和图例、表题和表注要让它们具备相对独立可读性。
- 同一组数据不要既做复杂图又做重复表，除非用户明确要求双重呈现。
- 结果段要概括图表的意义，不要逐行朗读。

### 表格

当读者需要精确数值或多变量并列时优先用表格。

检查点：

- 表题写的是“想让读者看见的点”，而不只是主题词；
- 单位是否齐全；
- 缩写是否在表内定义；
- 小数位和对齐是否专业一致；
- 技术细节能否放到表注，而不是挤进正文。

### 图形

当重点是趋势、对比、模式或流程时优先用图形。

检查点：

- 图是否能快速传达主要趋势；
- 组别是否容易区分；
- 坐标轴、符号和图例是否清楚；
- 如果图过于复杂，是否应改回表格；
- 图例是否足够支持独立阅读。

### 结果段与图表联动

围绕图表写正文时，按这个顺序组织：

1. 先讲最重要的结论，
2. 再引用图表，
3. 只挑关键数字强化这个结论，
4. 详细数值留给图表，
5. 负结果和对照结果在有意义时要主动交代。

## 审稿回复

当用户要根据评审意见修改论文、写回复信或 point-by-point response 时，按这里处理。

### 语气

- 具体，
- 尊重，
- 不防御，
- 即使评审说错了位置，也要找出真正触发问题的写作或论证缺陷。

### 回复模板

针对每条意见，优先按这个顺序写：

1. 简要复述评审关切，
2. 说明已做什么修改，
3. 指出修改出现在哪里，
4. 若不同意，给出清楚且礼貌的理由。

### 常见风险

- 把回复写成情绪表达而不是问题处理，
- 只回答表层意见，没有修掉附近真正的问题，
- 只说“已澄清”，却不说改了什么，
- 换投别刊时完全忽略原评审暴露出的写作缺陷。

## 综述文章扩展

只有在用户明确要写综述时才走这里，不要把原始研究论文写法生搬过去。

### 核心变化

目标不再是报告一项研究，而是围绕一个收束明确的主题综合一组文献。

### 工作流

1. 先把主题收窄到可控范围，不能只有大领域名。
2. 按问题、方法、机制、争议点或时间线来组织文献。
3. 明确分开“已知”“有争议”“未知”。
4. 结尾给出由综合分析自然推出的空白、建议或未来方向。

### 常见风险

- 主题太大，导致结构散。
- 只堆文献，不做综合判断。
- 章节顺序由读过的文献顺序决定，而不是论证结构决定。

## 大众摘要扩展

只有在用户明确要给非专业读者写摘要、新闻式说明或通俗介绍时才走这里。

### 核心变化

先给结论和意义，再给必要背景；尽量减少术语，把非专业读者真正需要知道的内容放到前面。

### 工作规则

- 开头先回答“为什么要关心这件事”；
- 专业术语能不用就不用，必须用时立即解释；
- 风险表达优先用绝对值、人数或更直观的整体数，而不是只给相对风险；
- 只保留支撑主信息的细节；
- 用短故事或具体场景承载信息，不要写成缩短版论文。

### 中文本土化提醒

- 中文大众摘要不宜照搬英文新闻 lead 的句法节奏，优先求直白、顺口和信息前置。
- 避免“显著”“机制”“模型构建”等研究者内部话语直接裸用。
- 数字最好配中文语境下更易理解的单位和比较方式。

### 常见风险

- 一上来先铺技术背景，主结论迟迟不出现；
- 术语和“科研黑话”太多；
- 只报相对风险，不给读者可理解的基线；
- 方法细节保留太多，淹没主信息。

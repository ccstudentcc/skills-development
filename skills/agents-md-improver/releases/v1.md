# `agents-md-improver` v1 发布说明

`agents-md-improver` 已在本仓库完成首个可复用版本发布。

这个技能面向一个长期但容易失控的问题：`AGENTS.md` 应该保留什么、放在哪一层、怎样在压缩后仍然保持执行边界清晰。它不以“增加规则”为目标，而是帮助使用者把最小、稳定、可验证的规则放到正确层级，并把不属于 `AGENTS.md` 的内容归位到更合适的文档。

## 本次发布包含的能力

- 判断一条信息是否适合进入 `AGENTS.md`。
- 在全局、仓库根、子目录三层之间选择最窄且稳定的落点。
- 将误放在 `AGENTS.md` 的 onboarding、架构说明或任务状态迁回 `README.md`、`ARCHITECTURE.md`、`SPEC.md` 或 `TASK_STATUS.md`。
- 在不削弱安全边界、作用域限制和验证要求的前提下，对冗长 `AGENTS.md` 做无损压缩。
- 基于重复记忆或 rollout 证据提炼可泛化规则，同时拒绝把仓库局部噪声误提升为全局规则。

## 适合处理的典型问题

| 场景 | `agents-md-improver` 的处理重点 |
| --- | --- |
| 根 `AGENTS.md` 逐渐变长 | 区分高信号执行约束与应迁出的背景内容 |
| 局部目录出现特殊约束 | 判断应该下沉到子目录还是保留在根层 |
| 多次出现同类失误 | 先检查证据是否足够稳定，再决定是否升格为长期规则 |
| 希望“压短”一个 `AGENTS.md` | 仅合并冗余表述，不压掉真正的边界条件 |
| 希望以后 agent 少看文件 | 保持文档分工，不把 `README.md` 和 `ARCHITECTURE.md` 再塞回 `AGENTS.md` |

## 技能工作流

这次发布沿用了一套偏保守的执行流程，先判断文档边界，再做最小修改。

| 阶段 | 关注点 | 产出方向 |
| --- | --- | --- |
| 指令链盘点 | 哪些规则已生效、哪些是继承而来 | 识别本层增量与上层约束 |
| 归属判断 | 这条信息该不该进 `AGENTS.md` | 确定全局、仓库根、子目录或非 `AGENTS.md` 落点 |
| 证据检查 | 这是稳定模式还是一次性噪声 | 决定升格、暂存观察或拒绝进入长期规则 |
| 最小改动 | 怎样改最短且不失真 | 增、删、迁移、压缩其一，而不是堆更多说明 |
| 最终审查 | 是否引入冲突、重复或边界漂移 | 保证安全边界、作用域和验证要求仍然清晰 |

## 验证结果摘要

本次发布附带了一轮更接近真实仓库使用方式的 comparative benchmark，重点验证的不是“能否运行”，而是“边界判断是否更稳定”。

### 覆盖场景

| Eval 类型 | 关注点 |
| --- | --- |
| `migration-boundary` | 临时迁移清单不应进入 `AGENTS.md` |
| `safety-conflict` | 子目录规则不能覆盖上层安全边界 |
| `insufficient-evidence` | 单条弱反馈不应升格成永久规则 |
| `global-vs-repo` | 仓库命令不应被抬到全局层 |
| `readme-boundary` | onboarding 和架构说明不应回流到 `AGENTS.md` |

### 对比结果

| 配置 | 通过情况 | Pass rate |
| --- | --- | --- |
| `with_skill` | `21/21` expectations passed | `1.0` |
| `without_skill` | `9/21` expectations passed | `0.4286` |

### 效率指标

| 指标 | `with_skill` | `without_skill` | 差值 |
| --- | --- | --- | --- |
| 单题平均耗时 | `76.2s` | `63.7s` | `+12.5s` |
| 单题平均 executor total tokens | `155,852` | `124,237` | `+31,615` |
| 单题最终回复长度 | `779` chars | `753` chars | `+26` chars |

`executor total tokens` 来自各 run 的 `timing.json.total_tokens`。其中有 1 个 `without_skill` run 没有正常 usage 记录，`timing.json.total_tokens` 为 `0`，因此没有纳入该均值。

`final response length` 指 benchmark 聚合中记录的 `final_response.md` 文本长度，不等同于 token 统计。

## 与 baseline 的主要差异

| 失真点 | baseline 常见问题 | v1 的改进方向 |
| --- | --- | --- |
| 全局与仓库分层 | 把 repo-specific 规则抬到全局层 | 把规则放回仓库层或子目录层 |
| 弱证据处理 | 把一次性观察写成永久 `AGENTS.md` 规则 | 改为记录在非持久文档，拒绝过早升格 |
| 安全边界继承 | 让局部 `AGENTS.md` 覆盖上层安全约束 | 保留上层安全边界，重写局部说明 |
| 文档边界 | 把 `README.md` / `ARCHITECTURE.md` 内容塞回 agent 指令 | 维持文档分工，避免 `AGENTS.md` 再次膨胀 |

## 当前边界

- `with_skill` 仍采用显式本地 skill 注入，不是 nested executor 的原生技能发现。
- 当前结果属于 `production-like comparative`，不是完全剥离仓库证据后的盲测。
- 新增的“基于记忆提炼全局规则”和“无损压缩”能力已经进入技能范围，但还没有单独的 fresh benchmark 专项结果。

## 仓库内的发布落点

- 运行时入口继续保留在 `SKILL.md`。
- 运行时细节和战术说明继续放在 `references/`。
- 人类可读的发布说明统一放在 `releases/`。
- 跨技能复用的评测、发布和任务协作文档统一放在仓库级 `docs/`。

这份 `v1.md` 对应的就是该技能的首发发布说明。
